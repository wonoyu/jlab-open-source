# API Integration Concepts

Understanding the architecture, patterns, and concepts behind the `wdev api` command for effective API integration in Flutter monorepos.

:::info Conceptual Documentation
This section provides in-depth explanations of how API integration works. For practical guides, see [How-to Guides](how-to-guides.md).
:::

## Overview

The `wdev api` command provides a comprehensive system for managing API services in Flutter monorepos. This section explains the underlying concepts and architecture.

---

## Architecture

### Service Layer Pattern

The API integration follows a clean architecture pattern with clear separation of concerns:

```
┌─────────────────────────────────────────────────────────────────┐
│                     Presentation Layer                           │
│  (BLoC, Provider, Riverpod - UI Components)                      │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                     Domain Layer                                 │
│  (Models, Interfaces, Business Logic)                            │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                   Data Layer                                     │
│  (Repositories, Data Sources, API Clients)                       │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                 External Services                                 │
│  (REST APIs, Databases, Caching)                                 │
└─────────────────────────────────────────────────────────────────┘
```

### Key Components

| Component | Purpose | Generated By |
|-----------|---------|--------------|
| **API Client** | HTTP communication with backend | `wdev api generate` |
| **Repository** | Data access abstraction | `wdev api integrate` |
| **Data Source** | Low-level API calls | `wdev api integrate` |
| **Model** | Domain data representation | `wdev api integrate` |
| **Interface** | Repository contract | `wdev api integrate` |

## Workflow

### 1. Service Creation

When you run `wdev api add`, the following happens:

```bash
wdev api add --name users
```

**Steps:**
1. Validates service name (snake_case required)
2. Creates OpenAPI spec file: `packages/api/lib/openapi/users.yaml`
3. Creates endpoint structure: `packages/api/lib/openapi/users/`
4. Generates example endpoints
5. Configures per-flavor URLs
6. Updates `config.yaml`

**Generated Files:**
```
packages/api/lib/openapi/
├── users/
│   ├── _index.json              # Service metadata
│   ├── get_users.json           # Example endpoint
│   └── post_user.json           # Example endpoint
└── users.yaml                   # OpenAPI spec
```

### 2. Endpoint Definition

Define endpoints using JSON files in the service directory:

```json
{
  "method": "get",
  "operationId": "getUsers",
  "summary": "Get all users",
  "responses": {
    "200": {
      "description": "List of users",
      "content": {
        "application/json": {
          "schema": {
            "type": "array",
            "items": { "$ref": "#/components/schemas/User" }
          }
        }
      }
    }
  }
}
```

### 3. Client Generation

Generate Retrofit clients from OpenAPI specs:

```bash
wdev api generate
```

**What happens:**
1. Parses OpenAPI specification
2. Generates Dart model classes
3. Creates Retrofit API interfaces
4. Generates Chopper-style annotations
5. Creates export file

**Generated Structure:**
```
packages/api/lib/src/generated/
├── users/
│   ├── users_api.dart           # Retrofit API
│   ├── users_api.chopper.dart   # Chopper annotations
│   └── users_api.freezed.dart   # Freezed models
└── export.dart                  # Barrel export
```

### 4. Service Injection

Inject the service into a feature with complete service layer:

```bash
wdev api integrate --service users --page home --feature dashboard --endpoint getUserById
```

**What happens:**
1. Generates repository implementation
2. Creates data source
3. Defines model classes
4. Sets up BLoC/state management
5. Registers in dependency injection

**Generated Structure:**
```
lib/src/features/dashboard/
└── home/
    ├── data/
    │   ├── repositories/
    │   │   └── users_home_repository.dart
    │   └── datasources/
    │       └── users_api.dart
    ├── domain/
    │   ├── models/
    │   │   └── users_model.dart
    │   └── interfaces/
    │       └── i_users_repository.dart
    └── presentation/
        └── blocs/
            └── home_users_bloc.dart
```

## Per-Flavor Configuration

API commands support per-flavor URLs for different environments:

```yaml
# config.yaml
flavors:
  dev:
    USERS_SERVICE_URL: https://users-dev.example.com
  staging:
    USERS_SERVICE_URL: https://users-staging.example.com
  prod:
    USERS_SERVICE_URL: https://users.example.com
```

**Usage:**
```bash
# Generate with specific flavor
FLAVOR=dev wdev api integrate --service users --page home --feature dashboard
```

## Caching Integration

### Drift Caching

Enable local caching with Drift:

```bash
wdev api integrate --service users --page home --feature dashboard --cache --cache-keys id
```

**Benefits:**
- Offline-first functionality
- Reduced API calls
- Faster data access
- Automatic sync

### PowerSync Integration

Enable offline-first with PowerSync:

```bash
wdev api integrate --service users --page home --feature dashboard --powersync
```

**Benefits:**
- Real-time sync
- Conflict resolution
- SQLite backend
- Cloud sync

## OpenAPI Specification

The `wdev api` command uses OpenAPI 3.0 specification:

```yaml
openapi: "3.0.0"
info:
  title: Users API
  version: "1.0.0"
paths:
  /users:
    get:
      operationId: getUsers
      summary: Get all users
      responses:
        "200":
          description: Successful response
```

## Environment Variables

Each service generates an environment variable:

| Service | Environment Variable |
|---------|---------------------|
| users | `USERS_SERVICE_URL` |
| orders | `ORDERS_SERVICE_URL` |
| products | `PRODUCTS_SERVICE_URL` |

**Usage in code:**
```dart
final serviceUrl = String.fromEnvironment('USERS_SERVICE_URL');
```

## Best Practices

### 1. Service Naming

- Use **snake_case**: `users`, `order_items`, `product_catalog`
- Avoid special characters
- Use plural nouns for collections

### 2. Endpoint Design

- Follow REST conventions
- Use meaningful operation IDs
- Include proper descriptions
- Define all response codes

### 3. File Organization

```
packages/api/lib/openapi/
├── users/              # Service directory
│   ├── _index.json    # Service metadata
│   ├── get_users.json # GET /users
│   └── `{id}`/          # Path parameters
└── users.yaml         # Generated spec
```

### 4. Caching Strategy

| Scenario | Caching Strategy |
|----------|-----------------|
| Static data | Full caching |
| Frequently changing | Time-based expiry |
| User-specific | Authenticated cache |
| Offline-first | PowerSync |

## Limitations

1. **No GraphQL support**: Only REST APIs supported
2. **Limited authentication**: Bearer token only
3. **No WebSocket support**: HTTP/1.1 only
4. **No request/response interceptors**: Custom implementation required

## Troubleshooting

### Common Issues

| Issue | Cause | Solution |
|-------|-------|----------|
| Service not found | Wrong name or path | Check `config.yaml` |
| Generation fails | Invalid OpenAPI spec | Validate with `wdev api add-endpoint --validate` |
| Integration fails | Missing feature | Create feature first |
| Caching doesn't work | No cache keys | Specify `--cache-keys` |

### Debug Commands

```bash
# Validate OpenAPI spec
wdev api add-endpoint --validate --source users/

# Preview integration
wdev api integrate --service users --page home --feature dashboard --dry-run

# Verbose output
wdev api generate --verbose
```
